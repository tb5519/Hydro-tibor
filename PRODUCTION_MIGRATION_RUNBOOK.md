# Hydro-tibor 正式迁移执行清单

## 一、当前阶段说明

- 本文是**正式迁移执行清单**，用于上线前按步骤核对与执行。
- **当前不执行迁移**：任何 dump、restore、rsync、切域名等操作均不在本文档编写时执行。
- **当前不连接服务器**：不在此阶段 SSH 或远程操作生产环境。
- **当前不导入数据**：不向任何环境导入 Mongo 或文件体。
- **当前不修改数据库**：不在此阶段对库做写入、删除或结构变更。
- 正式上线前，应由负责人按本文**逐条准备、校验、执行**，并保留操作记录与备份校验结果。

## 二、已完成的本地验证结论

以下结论来自本地/演练环境，**不代表生产已迁移**，但可作为正式迁移的技术依据：

- **UI v2** 已完成并通过人工验收（首页、题库、题目详情、在线编程等）。
- **badge-for-hydrooj 插件最小接入**已完成，加载与模板初始化正常。
- **用户详情页当前启用徽章展示**已完成（非多徽章方案）。
- **Mongo 旧数据可导入**：本地演练已验证导入路径与基本一致性。
- **`/data/file/hydro` 文件体必须单独恢复**：题面图、头像、附件等依赖该目录，**不可**仅依赖 `hydro-home.tar.gz`。
- **旧题 P1001** 已完成 **AC / WA** 的 judge 验证。
- **judge 链路可用**（演练环境）。
- **迁移技术路线可行**：Mongo + 文件体 + Web/Mongo 部署 + 插件与 judge 配置的组合路径已跑通。

## 三、正式迁移前必须准备的备份

正式迁移前必须齐备并可独立还原的备份与元数据（**缺一不可则不建议开迁**）：

1. **旧服务器 MongoDB dump**（全库或业务库，按旧站实际库名）。
2. **旧服务器 `/root/.hydro` 备份**（或等价 hydro 用户主目录配置，视旧部署而定）。
3. **旧服务器 `/data/file/hydro` 备份**（**完整文件体**，体积可能远大于 home 包）。
4. **`badge-for-hydrooj` 插件备份**（与仓库版本一致的可归档副本或固定 tag/commit）。
5. **`SHA256SUMS.txt`**（对 dump 包、文件体 tar、关键 tar.gz 等分别记录 sha256，**迁移前、迁移后各校验一次**）。
6. **旧服务器版本信息**（OS、Docker、Mongo、Node/Hydro 版本、内核）。
7. **旧服务器目录结构信息**（数据盘挂载点、Hydro 工作目录、上传目录实际路径）。
8. **旧服务器当前运行进程和端口信息**（避免新部署端口冲突，尤其与 ClassFlow 同机时）。

**强调：**

- **只备份 MongoDB 不够**；没有文件体则头像、附件、题面资源会大量失效。
- **`hydro-home.tar.gz` 不包含完整文件体**；不能替代 `/data/file/hydro`。
- **`/data/file/hydro` 必须单独备份、单独校验、单独恢复**。
- **正式迁移前必须校验 sha256**，与 `SHA256SUMS.txt` 一致后再传输/解压/restore。

## 四、正式迁移推荐窗口

建议：

- 选择**学生少用**的时间段（如深夜或周末低峰）。
- **提前通知**学生暂停提交、避免大文件上传与长时间比赛。
- **迁移期间冻结旧系统写入**（维护页或只读；至少停止写入类作业）。
- **ClassFlow 高峰期不要做迁移**（同机资源争用风险高）。
- **预留至少 2–3 小时**（备份校验、传输、restore、冒烟、回滚预留）。
- **旧服务器不要马上关停**：保留一段时间**只读**或冷备，便于对比与回滚。

## 五、正式迁移总体步骤

大步骤（顺序可按现场微调，但**备份与校验不得跳过**）：

- **Step 1**：冻结旧系统写入。
- **Step 2**：完整备份旧系统（Mongo + hydro 配置相关 + **`/data/file/hydro`** + 插件与元数据）。
- **Step 3**：校验备份（sha256、抽样解压、记录大小与文件数）。
- **Step 4**：部署新 Hydro-tibor **空系统**（或干净实例），版本与插件路径与演练一致。
- **Step 5**：确认新系统 **Web + Mongo** 正常（首页 HTML、管理入口、无异常 500）。
- **Step 6**：导入 MongoDB（见第八章草案；是否 `--drop` 由现场决定并**先备份新库空状态**）。
- **Step 7**：恢复 **`/data/file/hydro`**（见第九章草案）。
- **Step 8**：恢复必要 **static** 资源（若有独立静态资源或 CDN 外置，按旧站清单执行）。
- **Step 9**：配置 **badge 插件**（见第十章）。
- **Step 10**：配置 **judge 用户**（见第十一章；**不把密钥写入 Git**）。
- **Step 11**：启动 judge，**并发固定为 1**。
- **Step 12**：验证题目、用户、头像、附件、题面图片。
- **Step 13**：验证 **P1001** 正确代码 **AC**、错误代码 **WA**。
- **Step 14**：验证 **ClassFlow** 不受影响（页面、提交、资源占用）。
- **Step 15**：开放访问或**切换域名**（DNS / 反代切换前再跑一次冒烟）。
- **Step 16**：保留旧系统**只读**一段时间；确认无重大问题后再下线。

## 六、正式服务器资源策略

**当前服务器规划（文档级约定，以实际采购为准）：**

- 2 核 CPU  
- 2 GB 内存  
- 40 GB 硬盘  
- 3 Mbps 带宽  
- **同机运行 ClassFlow + Hydro**

**策略：**

- **先小规模上线**；不承诺比赛高并发。
- **judge 并发设为 1**；不追求 judge 速度，优先稳定。
- **不做比赛高并发**；大型比赛另议或另机。
- **优先保护 ClassFlow**（同机 Hydro 应限流、限并发、避免大任务占满内存）。
- 若内存**长期 80%+**：优先升级到 **4 GB**。
- 若磁盘剩余**低于 10 GB**：优先扩容到约 **80 GB**（或按增长预留）。
- 若提交**长期排队**且 CPU 有余量：再考虑 judge 并发 **2** 或 **CPU 升级**（每次只改一项并观察）。

## 七、正式部署时的 Docker 策略

- **Web + Mongo 使用 Docker**（与当前 hydro-tibor 开发/部署约定一致）。
- **judge 也可以 Docker**，但**并发固定 1**；初期不建议为 judge 单独拆机，降低运维面。
- **不暴露 judge 内部端口**到公网；仅 Hydro 与 judge 内网互通即可。
- **Docker volume 必须纳入备份策略**（尤其是上传与文件体所在卷）。
- **禁止**将 `docker compose down -v` 作为日常或迁移中的常规操作（**`-v` 会删卷**）。

## 八、正式迁移 MongoDB 命令草案

> **以下均为草案，不执行。** 正式执行时替换 `<...>` 为真实主机、路径、库名；**不要把数据库密码写入 GitHub 或本仓库文档**。

```bash
# 1) 在操作机上创建正式迁移前备份目录（示例路径，请替换）
mkdir -p "<BACKUP_ROOT>/hydro-migrate-<DATE>"
cd "<BACKUP_ROOT>/hydro-migrate-<DATE>"

# 2) 在旧服务器或能访问旧 Mongo 的机器上执行 mongodump（示例）
# mongodump --uri="<MONGODB_URI>" --out="./mongo-dump-<DATE>"
# 或使用 --db <DB_NAME> 等参数按旧站实际调整

# 3) 将 dump 目录打包并生成校验和（示例）
# tar czf "mongo-dump-<DATE>.tar.gz" "./mongo-dump-<DATE>"
# sha256sum "mongo-dump-<DATE>.tar.gz" >> SHA256SUMS.txt

# 4) 下载到本地或新服务器后校验 sha256
# sha256sum -c SHA256SUMS.txt   # 或只对 mongo 包单独校验

# 5) 在新服务器将 dump 拷入可访问路径后 mongorestore（示例）
# 是否使用 --drop：若新库为空且确认目标正确，可考虑；若有共存需求则禁用 --drop 并事先评估冲突
# mongorestore --uri="<NEW_MONGODB_URI>" --drop "./mongo-dump-<DATE>/<DB_DIR>"
# 或 mongorestore --nsInclude="<DB_NAME>.*" ...

# 6) 验证 collection 数量（示例，使用 mongosh）
# mongosh "<NEW_MONGODB_URI>" --eval 'db.getSiblingDB("<DB_NAME>").getCollectionNames().length'
# 并与旧站导出前的统计或清单对比
```

**强调：**

- 正式执行时必须替换**真实路径与连接串**；连接串中的密码**仅**出现在服务器侧环境变量或密钥管理，**不进 Git**。
- **`--drop` 会删除目标库已有集合**，使用前必须二次确认连接的是**新环境**而非误连生产旧库。

## 九、正式恢复 `/data/file/hydro` 命令草案

> **以下均为草案，不执行。**

```bash
# 1) 旧服务器打包（在旧机、路径需与真实部署一致）
# sudo tar -C /data/file -czf "hydro-file-hydro-<DATE>.tar.gz" hydro
# 或 tar 源路径按实际 /data/file/hydro 调整

# 2) 生成 sha256
# sha256sum "hydro-file-hydro-<DATE>.tar.gz" >> SHA256SUMS.txt

# 3) 下载到本地或新服务器后校验
# sha256sum -c SHA256SUMS.txt

# 4) 在新服务器解压到 Docker volume 对应宿主目录（示例，需对照 compose volume 挂载点）
# sudo tar -C "<HOST_PATH_TO_FILE_VOLUME>" -xzf "hydro-file-hydro-<DATE>.tar.gz"

# 5) 确认目录大小与文件数（示例）
# du -sh "<HOST_PATH_TO_FILE_VOLUME>/hydro"
# find "<HOST_PATH_TO_FILE_VOLUME>/hydro" -type f | wc -l

# 6) 业务验证：随机抽用户头像、题目附件、题面图片、评测样例依赖文件是否 200/可读
```

**强调：**

- **不要只恢复 `hydro-home.tar.gz`** 当作文件体完备。
- **不要把**旧站 **`config.json` / `Caddyfile` / `mount.yaml`** 等**无脑覆盖**新环境已验证的配置；应逐项 diff 后合并。
- **不要恢复**旧站垃圾文件如 **`.git` / `.bak` / `.save`** 等到数据卷。

## 十、badge 插件正式配置

- **已接入插件路径**：`addons/badge-for-hydrooj`（与仓库一致）。
- **正式启动时 `HYDRO_ADDONS_JSON` 必须包含**（顺序与路径按部署环境调整，以下为最小示例）：

```json
["@hydrooj/ui-default","/workspace/addons/badge-for-hydrooj"]
```

**注意：**

- **不能漏掉 `@hydrooj/ui-default`**，否则前端/主题链路异常时可能出现**非 HTML 响应**（如 JSON）等风险。
- **当前只支持「当前启用徽章」展示**；**多徽章功能暂不做**，且**不应在正式迁移窗口内**混入多徽章开发。
- 旧数据中的 **`badge` / `userBadge`** 已在本地演练中确认存在；正式迁移后仍需抽样核对。
- 正式迁移后需验证（已登录前提下）：`/mybadge`、`/manage/badge`、`/badge/add` 权限与页面正常。

## 十一、judge 正式配置

- **judge 并发设为 1**（与资源策略一致）。
- 若无 **judge 用户**：需按 Hydro 文档创建用户并 **`setJudge`**。
- **`judge.yaml` 不允许提交到 Git**；仅存在于服务器或密钥管理。
- 本地演练曾出现 **uid=36** 等仅为**实验环境**示例，**不代表正式环境**；正式环境按实际库内用户 ID 创建或校准。
- 需验证：**judge 与 Hydro 连接**、队列消费、结果回写。
- 需验证：**P1001 正确代码 AC**、**P1001 错误代码 WA**。

## 十二、正式迁移验收清单

上线或切流量前逐项打勾（可打印为检查表）：

1. 首页能打开（返回 HTML，非异常 JSON）。  
2. 题库页能打开。  
3. 旧题能打开（抽样 + 含 P1001）。  
4. 用户头像正常。  
5. 题面图片正常。  
6. 附件可访问。  
7. **P1001** 正确代码 **AC**。  
8. **P1001** 错误代码 **WA**。  
9. 排行榜正常。  
10. 用户页正常。  
11. 徽章展示正常（用户详情「当前启用」路径）。  
12. `/mybadge` 正常（登录态）。  
13. `/manage/badge` 管理页权限正常（管理员）。  
14. **ClassFlow** 正常。  
15. 服务器内存和 CPU 正常（无持续打满）。  
16. **磁盘剩余大于 10 GB**（或按扩容策略已满足）。  
17. **没有 500**（核心路径抽样 + 日志无大面积 Error）。  
18. 浏览器 **Console** 无严重红色报错（首页、题面、提交页）。

## 十三、回滚方案

若正式迁移失败或验收不达标：

1. **不要删除旧服务器**。  
2. **不要删除旧备份**。  
3. **停止新 Hydro**（停止容器/服务，而非删卷）。  
4. **恢复新服务器迁移前 Mongo 备份**（按第八章反向操作，使用事先保留的 restore 前快照）。  
5. **恢复新服务器迁移前 `/data/file/hydro`**（或整卷回滚方案）。  
6. **域名切回旧服务器**（DNS / 反代回滚）。  
7. **保留失败现场日志**（Hydro、Mongo、反代、系统日志）。  
8. **不要执行 `docker compose down -v`**（避免误删卷）。  
9. **不要在无备份、无校验的情况下重试全量迁移**。

## 十四、禁止事项

- 不允许**未备份**就迁移。  
- 不允许**只导 MongoDB** 而忽略 **`/data/file/hydro`**。  
- 不允许**忘记 `/data/file/hydro`** 的校验与恢复。  
- 不允许**覆盖**生产 `.env` 为未审核内容。  
- 不允许**覆盖** `docker/judge/judge.yaml` 等到 Git 中受控路径的敏感文件（且敏感文件本不应进仓库）。  
- 不允许**覆盖**已验证的 Docker Compose 配置而不留回滚副本。  
- 不允许把**密码、Token、私钥**写入 GitHub 或本仓库。  
- 不允许在 **ClassFlow 使用高峰期**做迁移。  
- 不允许迁移后**马上关停**旧服务器（应保留只读观察期）。  
- 不允许把**多徽章功能开发**混入正式迁移窗口。
